# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Test

The repository uses a [Nuke](https://nuke.build/) build (`build/_build.csproj`) wrapped by `build.ps1` / `build.cmd` / `build.sh`. The wrapper scripts will download the SDK pinned in `global.json` (currently .NET 10) if a matching `dotnet` is not on PATH.

- Full CI pipeline locally: `./build.cmd Compile Test Pack Publish` (the GitHub workflow runs the same line).
- Individual Nuke targets: `Clean`, `Restore`, `Compile`, `Test`, `Pack`, `Publish`. `Publish` is gated to non-Windows + main/tag builds.
- Plain dotnet usage works too — `dotnet build Test262Harness.sln`, `dotnet test test/Test262Harness.Tests/Test262Harness.Tests.csproj`.
- Run a single NUnit test: `dotnet test test/Test262Harness.Tests --filter "FullyQualifiedName~CanLoadFromGitHub"`. Note that `CanLoadFromGitHub` performs a real network download of the test262 archive.
- Build outputs and test results go to `artifacts/` (`<UseArtifactsOutput>true</UseArtifactsOutput>` is set in `Directory.Build.props`).

`TreatWarningsAsErrors=true` and `AnalysisLevel=latest-Recommended` are enforced repo-wide; build failures will trip on new analyzer warnings.

## Solution Layout

Five projects in `Test262Harness.sln` form a layered pipeline:

1. **`src/Test262Harness`** (library, `netstandard2.0;net8.0`) — the core. Parses test262 files and exposes them as a stream.
   - `Test262Stream` enumerates `.js` test files from a `Zio` `IFileSystem`. Construct via `FromDirectory`, `FromZipArchive`, or `FromFileSystem`. `Test262StreamExtensions.FromGitHub(commitSha)` downloads `https://github.com/tc39/test262/archive/{sha}.zip` into the temp directory and wraps it.
   - `Test262File.FromStream` parses the YAML frontmatter (`/*--- ... ---*/`) and yields one or two instances per file (script files produce both a sloppy and a `"use strict"`-prefixed variant when `GenerateInverseStrictTestCase` is on; modules yield exactly one).
   - `Test262Runner` is an opt-in parallel executor — callers supply `Execute`, `ShouldThrow`, `IsParseError/IsResolutionError/IsRuntimeError`, and `IsIgnored` callbacks via `Test262RunnerOptions`, and it sorts results into `TestExecutionSummary` buckets (allowed/disallowed × success/failure/false-positive/false-negative).

2. **`src/Test262Harness.Generator`** (library, `net8.0`) — converts a `Test262Stream` into generated NUnit source files using Fluid (Liquid) templates embedded from `Templates/{TestFramework}/{Test262Test,Tests}.liquid`. Driven by `TestSuiteGenerator.Generate`, which groups tests per top-level test262 subdirectory and emits one `.generated.cs` per subdirectory plus a shared bootstrap file. `TestFramework` enum currently only has `NUnit` — adding a framework means dropping a parallel template directory.

3. **`src/Test262Harness.Console`** (`net8.0`, packed as a `dotnet tool` named `test262`) — Spectre.Console CLI. Sole command is `generate`, which loads `Test262Harness.settings.json` from the working directory (overrides via CLI flags), builds a `Test262Stream` from either `SuiteGitSha` or `SuiteDirectory`, and runs the generator. End users add this tool to their test project (`dotnet tool add Test262Harness.Console`) and run `dotnet test262 generate` — see README for the full consumer workflow.

4. **`src/Test262Harness.Runner`** (exe, `net8.0`, not packed) — secondary runner using `JavaScriptEngineSwitcher` (Jint / Jurassic / NiL). Not the primary product; the README points users at downstream projects (Jint, esprima-dotnet, acornima) that consume the generator and supply their own engine integration.

5. **`test/Test262Harness.Tests`** — NUnit + FluentAssertions. Globally enables `using NUnit.Framework;` and `using FluentAssertions;` via `<Using>` items in the csproj.

## Conventions

- **Central package management**: all versions live in `Directory.Packages.props`. Add a `PackageVersion` there before adding a `PackageReference` in a csproj (no version on the reference).
- **Assembly signing**: `Test262Harness.snk` is checked in but `<SignAssembly>` is `false` — leave it that way unless explicitly changing the signing strategy.
- **GitHub Actions workflows** under `.github/workflows/` are auto-generated by Nuke (`[CustomGitHubActions]` on `Build.CI.GitHubActions.cs`). Don't hand-edit `build.yml` / `pr.yml`; regenerate via `nuke --generate-configuration GitHubActions_build --host GitHubActions`.
- The `Test262Harness.settings.json` schema (consumed by `GenerateCommand.Settings` / `TestSuiteGeneratorOptions`) is documented in the README's configuration table; exclusion entries become `[Ignore]` attributes on emitted NUnit tests, with esprima-style `(default)` / `(strict mode)` suffixes and glob patterns both supported.
